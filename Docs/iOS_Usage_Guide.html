iOS 打包与本地资源部署说明

这是给 Mac + Unity + Xcode + iPhone 的现场使用说明。先拉最新 git，再打 iOS AssetBundle，再把资源部署到本机静态服务器，最后导出 Xcode 工程跑到真机上。

热梗版一句话： 先让手机找到 Mac 的资源服务器，不要把 127.0.0.1 塞进真机。 点这里就是刚才那个资源网页地址
Unity 2022.3.62f1 平台 iOS 资源 AssetBundle Xcode 26.3

当前可用地址

这两个地址是现在最重要的入口。先点资源根地址，再点 iOS manifest。手机和 Mac 必须在同一个局域网。

资源根地址 http://192.168.1.6:8000/theweepingswan/1.0.0
iOS manifest http://192.168.1.6:8000/theweepingswan/1.0.0/iOS/iOS
这是什么 这是 UTAGE 资源服务器，不是官网吗。iPhone 运行时会先来这里拿平台目录，再往下取具体 AssetBundle。

推荐流程

按这个顺序走，最稳。少走回头路，少把大包白打一遍。

1

拉最新 git

先同步主分支，避免拿旧配置去打包。

2

打 iOS 资源包

生成 BuildAssetBundles/iOS。

3

部署到本地服务器

复制到 Deploy/AssetServer/theweepingswan/1.0.0/iOS。

4

导出 Xcode

在 Builds/iOS/TheLamentingGeese-iOS 里跑到真机。

开始前先拉最新 git

如果你在另一台机器上开工，这一步一定先做。仓库里有本地改动时，先看状态再拉。

  1. 进入仓库根目录。
  2. 先看当前分支和本地改动。
  3. 拉取远端 main 的最新提交。
  4. 如果有冲突，先处理干净，再继续 Unity。
git 同步命令 
git status --short --branch
git fetch origin
git pull --ff-only origin main
git rev-parse HEAD
提醒 如果本地已经有未提交改动，先 stash 再 pull，别让资源文件和配置文件夹在一起打架。

iOS AssetBundle 打包

这一步只负责生成资源包，不会导出 Xcode。打完之后你会看到 BuildAssetBundles/iOS/iOS 和 iOS.manifest。

  1. 打开 Unity 项目，确认目标平台已经切到 iOS。
  2. 如果你在编辑器里要看远程资源，先切 Codex/Asset Runtime Mode/Remote AssetBundles，再把目标切到 iOS。
  3. 执行 Codex/AssetBundles/Build iOS。
  4. 等 BuildAssetBundles/iOS 完成，确认顶层 iOS 与 iOS.manifest 都存在。
Unity 菜单 
Codex/Asset Runtime Mode/Remote AssetBundles
Codex/Asset Runtime Mode/Remote Target/iOS
Codex/AssetBundles/Build iOS
产物检查 
ls -lh BuildAssetBundles/iOS/iOS BuildAssetBundles/iOS/iOS.manifest
find BuildAssetBundles/iOS -maxdepth 2 -type d -print | sort

部署到本地静态服务器

把打好的 iOS 资源包复制到本机 HTTP 目录。这个目录是给手机和 Xcode 运行时读取的。

  1. 先执行部署脚本，把 BuildAssetBundles/iOS 复制到 Deploy/AssetServer/theweepingswan/1.0.0/iOS。
  2. 确认 Deploy/AssetServer/theweepingswan/1.0.0/iOS/iOS 可见。
  3. 启动 HTTP 服务，推荐监听 0.0.0.0:8000，这样局域网里的其他设备也能访问。
  4. 如果你改了 Mac 的 IP，只改本地覆盖文件，不用重打资源包。
部署命令 
Tools/Deployment/prepare-asset-bundle-deploy.sh --source-root BuildAssetBundles --platforms iOS --version 1.0.0
启动本地服务器 
Tools/Deployment/start-local-asset-server.sh --port 8000 --bind 0.0.0.0 --foreground
本机 IP 现在是 192.168.1.6。手机要访问的是这个 IP，不是 127.0.0.1。

Xcode 导出与真机安装

这里会生成可打开的 Xcode 工程。目标工程目录是 Builds/iOS/TheLamentingGeese-iOS。

  1. 在 Unity 里执行 Codex/Build iOS Xcode Project。
  2. 导出后打开 Builds/iOS/TheLamentingGeese-iOS/Unity-iPhone.xcodeproj。
  3. Xcode 里选择你的 iPhone 真机。
  4. 第一次装包如果提示证书不受信任，先去 iPhone 设置里信任开发者证书。
  5. 如果手机上是白屏或黑屏，先看控制台有没有资源服务器请求。
Unity 菜单 
Codex/Build iOS Xcode Project
命令行导出 
/Applications/Unity/Hub/Editor/2022.3.62f1/Unity.app/Contents/MacOS/Unity \
  -quit -batchmode \
  -projectPath /path/to/TheLamentingGeese \
  -executeMethod CodexIOSBuild.BuildXcodeProject

手机跑起来时应该看到什么

这部分是判断“到底有没有真的启动”的关键。

  1. App 启动后，控制台应该先看到 Unity 初始化信息，然后看到 CODEX_ASSET_SERVER_BOOTSTRAP。
  2. 接着应该能看到对 http://192.168.1.6:8000/theweepingswan/1.0.0/iOS/iOS 的请求。
  3. 如果资源很多，第一屏可能会有等待页或加载状态。
  4. 如果你什么都看不到，先别急着重打包，优先看证书、网络和服务器日志。
最常见的误会 真机能装上，不代表它已经拿到资源。要看它有没有真正打到本机服务器。

看起来没反应时怎么查

这是给“装上了但像没开”的情况准备的。按这个顺序查，一般能很快定位。

1. 先确认证书是不是被信任

如果 Xcode 日志里出现 “Developer App Certificate is not trusted”，去 iPhone 的设置里把开发者证书信任掉，再重新跑一次。

2. 先确认服务器能从手机访问

在手机 Safari 里打开这个地址：

http://192.168.1.6:8000/theweepingswan/1.0.0/iOS/iOS

能看到下载响应，说明局域网链路是通的。

3. 如果 App 启动但没加载内容

看 Xcode console 里有没有 CODEX_ASSET_SERVER_BOOTSTRAP、CODEX_ASSET_BUNDLE_ALIAS_MAP、DownloadManifestAsync 之类的日志。

  • 只有 bootstrap，没有 manifest 请求，多半是启动加载链路还没跑通。
  • 连 bootstrap 都没有，说明更早的启动阶段有问题。
  • 如果提示证书问题，先处理证书，不要先怀疑资源包。
4. 如果 Mac 的 IP 变了

改这个文件即可：

Deploy/asset-server-url-iOS.txt

然后重新导出 Xcode 工程，不用重打 AssetBundle。

5. 如果你只想编辑器里本地跑

切到 Codex/Asset Runtime Mode/Editor Local Assets，然后编辑器里走本地资源，不走远程 HTTP。
